The Big Picture

These three ideas build on each other. Reasoning lets AI think before it answers. Structured outputs let us trust the shape of that answer. Agents combine both — an AI that thinks about a problem, decides on an action, takes that action using a tool, and repeats until the job is done.

Reasoning — Teaching AI to "Think First"

AI Doesn't Actually "Think" — It Predicts

We learned earlier that an AI model is fundamentally a next-word prediction machine. Given some text, it predicts the most likely next word, then the next, then the next. That's it. There's no little person inside "thinking" about your question.

Simple Analogy — The World's Best Autocomplete

Imagine the autocomplete on your phone, but trained on basically the entire internet. When you type "The capital of France is", it predicts "Paris" — not because it "knows" geography in the way a person does, but because it has seen that pattern of words millions of times.

This works great for simple facts. But what about a maths problem, or a question with several steps? If the AI just blurts out the first answer that "sounds right" based on patterns, it often gets it wrong.

The Problem With Jumping Straight to an Answer

Imagine asking someone a tricky multi-step maths problem and demanding they shout out the answer immediately, with no time to think, no scribbling on paper. They'd probably get it wrong — not because they're bad at maths, but because they didn't get to work through the steps.

Early AI models had exactly this problem. Asked a multi-step question, they'd generate an answer immediately — and often it was wrong, even if the model "knew" all the individual facts needed to solve it.

The Discovery: "Think Step by Step"

Researchers found something surprisingly simple: if you just add the words "Let's think step by step" to the end of a prompt, the model's answers improve dramatically — sometimes from getting a problem wrong almost every time to getting it right almost every time.

Why Does This Work?

The AI generates text one word at a time, and each new word is influenced by everything written before it — including its own previous words. If the AI first writes out its reasoning ("First, I need to find X. Then I can use X to calculate Y..."), each step becomes part of the context for the next step.

It's like the AI is writing out its working on scrap paper. By "thinking out loud" in text, it gives itself a scaffold to build the correct answer on, instead of trying to leap straight to the final answer in one go.

This Has a Name: Chain-of-Thought (CoT) Prompting

This technique — asking the model to reason through a problem step-by-step before giving a final answer — is called Chain-of-Thought prompting, often shortened to CoT. It's one of the simplest and most effective prompting techniques ever discovered.

From a Trick... to How Models Are Built

"Think step by step" started as a prompting trick that users had to remember to type. But it worked so well that AI companies started asking: what if the model just always did this automatically, without being told?

This led to what are now called reasoning models. During an extra training stage (called reinforcement learning, applied after the main training), these models are specifically trained to pause and reason through problems internally before producing a final answer — without the user needing to ask for it.

Key Takeaway

Today, almost all modern AI models do some form of reasoning, whether or not they're explicitly marketed as "reasoning models." Some show you their reasoning steps, others do it behind the scenes and only show you the final answer. Either way, the core idea is the same: better answers come from thinking before answering, and that "thinking" is really just generating helpful intermediate text.

Structured Outputs — Getting AI to Answer in a Predictable Shape

The Problem: AI Is Unpredictable by Nature

Imagine you're building an app. A user types in a product description, and you ask an AI to extract the product name, price, and category so your app can store them in a database.

You might write a prompt like: "Extract the product name, price, and category from this text, and return it as JSON."

This often works! But sometimes, the AI responds with something slightly different than you expected:

What you wantedWhat you sometimes get
{"name": "...", "price": 29.99, "category": "..."}Sure! Here's the JSON you asked for: {"name": "...", ...} (extra text before the JSON)
"price": 29.99 (a number)"price": "£29.99" (a string, with a currency symbol)
"category": "Electronics""category": "electronics" or "Electronic Devices" (inconsistent wording)
Valid JSONAlmost-valid JSON with a missing comma or bracket

Why Does This Happen? "Non-Determinism"

A traditional computer program is deterministic — give it the same input, it always produces the exact same output, every time, forever. A calculator always says 2+2=4.

AI models are non-deterministic — give them the same prompt twice, and you might get two slightly different responses. We actually learned why in a previous lesson: remember temperature? Even at low settings, there's an element of probability in which word gets picked next. Multiply that across hundreds of words, and small variations add up.

Simple Analogy — Asking a Person vs. a Vending Machine

A vending machine is deterministic: press B4, get the same chocolate bar every single time. A person, however, might describe the same chocolate bar slightly differently each time you ask — "the Mars bar," "a chocolate bar with caramel," "B4's snack." Same underlying thing, different wording each time.

AI is like the person, not the vending machine. It "knows" what you want, but how it phrases or formats the response can vary.

Why This Is a Real Problem for Developers

If your app's code expects price to always be a number, but the AI sometimes returns it as a string like "£29.99", your code will crash or behave incorrectly. Multiply this by thousands of requests per day, and even a 1% failure rate becomes a significant number of broken requests.

Solution 1: Validation

The first line of defence is validation — checking the AI's output against a strict definition of what's "allowed" before your app uses it. If the output doesn't match, you reject it and can ask the AI to try again.

A popular tool for this in JavaScript/TypeScript is called Zod. With Zod, you define a "schema" — a description of exactly what shape and types your data should be — and Zod checks the AI's output against it.

Simple Analogy — A Bouncer With a Checklist

Think of validation as a bouncer at a club door holding a checklist: "Is this a number? Is this field present? Is this one of the allowed categories?" If the AI's output passes the checklist, it's let through to your application. If not, it's rejected — and your code can ask the AI to try again, or handle the error gracefully.

Solution 2: Provider Tooling (Built-in Structured Outputs)

The second approach is even more powerful: many AI providers (like OpenAI) now offer a feature often called "structured outputs" built directly into their API. Instead of just hoping the AI follows your formatting instructions, you give the provider an exact schema, and the provider guarantees the response will match it.

This works very differently from prompting. Rather than relying on the AI to "remember" to follow the format, the underlying system mathematically restricts which words the AI is even allowed to generate next — so it's physically incapable of producing invalid JSON.

Key Takeaway

Prompting alone ("please return JSON") is the weakest approach — it usually works, but "usually" isn't good enough for production software. Validation (like Zod) catches problems after the fact. Provider structured-output tools prevent the problem before it happens, by constraining what the model can generate. In real applications, teams often use both together.

AI Agents — When AI Can Take Action

Section 3 · 50 minutes

What Is an AI Agent?

So far, everything we've discussed is about AI generating text — answers, code, JSON. But text alone can't check today's weather, search a database, send an email, or browse a website. For that, the AI needs to be connected to tools that can actually do things in the real world.

A Simple Definition

An AI agent is: giving an AI model the ability to take certain actions in order to achieve a goal. The AI doesn't just respond with text — it can decide "I need to look something up," actually look it up using a tool, see the result, and then decide what to do next.

Agents vs. Workflows — What's the Difference?

This distinction trips a lot of people up, so let's be very clear:

WorkflowAgent
Who decides the steps?A human, in advanceThe AI, in the moment
Order of operationsFixed: Step 1, then Step 2, then Step 3, alwaysFlexible: the AI decides what to do next based on what just happened
Example"Always: receive email → check spam score → if low, forward to support team""Figure out the best way to answer this customer's question — you can check our docs, look at their order history, or ask a follow-up question, in any order"
Simple Analogy — A Recipe vs. a Chef

A workflow is like a recipe: do step 1, then step 2, then step 3, in that exact order, every time. It's predictable and reliable, but inflexible — if something unexpected happens, the recipe doesn't adapt.

An agent is like a chef: given a goal ("make a tasty dinner with what's in the fridge"), the chef looks at what's available, decides what to make, checks on it while cooking, and adjusts as they go. More flexible, but also less predictable — and a less experienced chef might make worse decisions.

What Does an Agent Need?

For an AI to act as an agent, it needs two things — both of which we've already covered in this lesson!

  • Reasoning — the ability to look at a situation and decide what to do. ("The user wants to know the weather. I don't know today's weather. I should use a weather tool.")
  • Structured outputs / tool calling — the ability to specify, in a reliable format, exactly which tool to use and what information to give it. ("Call the get_weather tool with location: 'London'.")

Without reasoning, the AI can't decide what to do. Without structured outputs, even if it decides "I should check the weather," there's no reliable way for the surrounding code to understand and execute that decision.

The ReAct Pattern: Reasoning + Acting

One of the most common and well-known patterns for building agents is called ReAct, which stands for Reasoning + Acting.

The ReAct pattern works in a repeating loop:

1
ThoughtThe AI reasons about the current situation: "What do I know? What do I still need to find out?"
2
ActionThe AI decides to use a tool, and specifies it in a structured format: "Call search_database(query='...')"
3
ObservationThe tool runs (in your code, not inside the AI) and the result is given back to the AI
4
RepeatThe AI reasons again with this new information — until it decides it has enough to give a final answer
Simple Analogy — Detective Solving a Case

A detective doesn't solve a case in one leap. They think ("I need to know where the suspect was at 9pm"), take an action (interview a witness), get an observation (the witness's answer), then think again based on that new information ("Now I should check the CCTV footage"), and so on — looping until they've solved the case.

A ReAct agent works the same way: think, act, observe, repeat, until the goal is achieved.

What Does This Look Like in Practice?

Here's the key thing to understand: the AI model itself never actually runs any code or calls any tool directly. The AI only ever generates text — that's all it can do. The "magic" is in how the surrounding program is written:

1
Define toolsYour code defines simple functions — e.g. get_weather(city), search_products(query) — and describes them to the AI (name, what it does, what input it needs)
2
AI decidesThe AI, given the user's goal and the list of available tools, generates structured output saying which tool it wants to call and with what input
3
Your code executesYour program reads that structured output, actually runs the real function, and gets a real result
4
Feed it backYour code sends that result back to the AI as new context, and asks it to continue
Key Takeaway

An "agent" isn't magic — it's a loop, written in normal code, that repeatedly asks the AI "given everything so far, what should happen next?", runs whatever the AI decides, and feeds the result back in. The AI provides the decision-making; your code provides the actual capability to act.

A Riskier Approach: Code Execution Agents

The ReAct pattern above limits the AI to choosing from a small, pre-defined list of safe tools that your code controls. But there's another, more powerful approach: code execution agents, where instead of picking from a fixed list of tools, the AI writes its own code on the fly and that code gets executed.

Why This Is Riskier

With a fixed set of tools, you know exactly what's possible — the AI can only call get_weather or search_products, nothing else. With code execution, the AI can write any code, which means it could (intentionally or by mistake) do something harmful: delete files, access data it shouldn't, make unintended network requests, or get stuck in an infinite loop.

This is powerful — it lets the agent solve problems nobody anticipated in advance — but it's also much harder to make safe. If you're new to agents, stick with the ReAct pattern and a small set of carefully controlled tools until you're very comfortable with how agents behave. Code execution agents are an advanced topic for later.
